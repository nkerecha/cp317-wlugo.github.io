1. Overview

WLU Go is a web application that allows users to view, rate, and review the courses offered at Wilfrid Laurier University (WLU). It would help students get an idea of what a course is like by reading reviews and different rating parameters. As of now, there is no dedicated way for WLU students to get a thorough understanding of the courses offered other than through word of mouth. We want WLU Go to be a one-stop universal web application that is easily accessible and provides information on any WLU course with just a quick course code search. 

1.1 Purpose and Scope

This document outlines the requirements of WLU Go, a web application that will allow WLU students to leave course reviews and ratings. This document is targeted towards the developers and to any users who want a deeper understanding of the application and its uses.

The scope of the project is a fully functional web-based application that allows Wilfrid Laurier University students access to information on every course offered at the university. The objective of WLU Go is to help students gain a better understanding of WLU courses based on the experiences of others, desired birdness (difficulty), enjoyability, and usefulness. Through the use of this application, students will have an easier time deciding which courses to take.

1.2 Intended Audience

The intended audience for this web application is future and current Wilfrid Laurier University students who are looking to get some insight on the courses which they will potentially take or want to leave some knowledge about their experience with their respective course. In addition, the users of this web application are expected to have a basic understanding of how to use a modern website and internet browser. It will be helpful but not necessary if users have previous experience with other community-like social platforms (e.g. Reddit or Facebook). With this prior expertise, users will be comfortable using some of the features WLU Go has to offer, such as a comment system, rating network, and feedback loop.

1.3 Constraints

Multiple users must be able to access the website at the same time. To ensure that the web application is reliable, speedy, and smooth, reviews and user comments should be updated within a 5-second window of when they were originally sent by the student. 

It is assumed that there will be no cost constraints. There might be some constraints in the development cycle of this web application. This includes the potential reliability constraints depending on which Frameworks or APIs are used. In addition, there can be possible time constraints with the development cycle with the scrapping/gathering of large data and importing those figures into the database. 

Further, select members of the team will be expected to learn and use the technologies which will be deployed during this project. Members might have to gain some additional knowledge on some front-end or back-end technology stacks that will be used for the creation of the web application. This might add to the overall time constraint of this project, which needs to be finished in July/August.

1.4 Assumptions and Dependencies

It is assumed that all users will have some sort of modern internet browser (e.g. Google Chrome, Microsoft Edge, Mozilla Firefox, etc.). This web application can be accessed both via a computer or a mobile device. 

Users are also expected to have a stable internet connection for the entire duration of their session. As well, WLU Go should be compatible with any modern operating system (e.g. Windows 10, Chrome OS 90, Mac OS 10, etc.).

3. Definitions

3.1 WLU: Wilfrid Laurier University. The educational institution whose courses will be evaluated by the users.

3.2 API: Application Program Interface. External software packages are the intermediary between applications used in building WLU Go.

3.3 GUI: Graphical User Interface. This is the interface with which the user interacts. It consists of traditional navigation buttons, rating bars, and relevant information.

3.4 SRS: Software Requirements Specifications. This is the document that has all requirements and standards contained.

3.5 DDMS: Database Management System. This is the software that is used to store and retrieve users’ data while considering appropriate security measures.

3.6 FMS: File Management System. This is the process of administering a system that correctly handles all digital data/information/files.

3.7 HTTPS: Hypertext Transfer Protocol Secure: Hypertext Transfer Protocol Secure is an extension of the Hypertext Transfer Protocol. It is used for secure communication over a computer network and is widely used on the Internet.

3.8 Birdness: The level of course difficulty experienced by the student reviewer on a scale of 1-5 with 1 being very difficult and 5 being very easy.

3.9 Usefulness: The level of course usefulness and applicability experienced by the student reviewer in their field on a scale of 1-5 with 1 being very useless and 5 being very useful.

3.10 Enjoyability: The level of course enjoyability experienced by the students of 1-5 with 1 being very unenjoyable and 5 being very enjoyable.

3.11 Grade Received: The grade that the user received after completing the course. This will be an optional input for all users. 

3.12 Professor: The full name of the professor who taught the course. This will be an optional input for all users. 

3.13 Course Name/Course Code: The course code or the name of the course.

3.14 Course Description: The description of the course that Wilfrid Laurier University provides.

3.15 Guest User: A guest user is someone who uses the application to view reviews. 

3.16 Review: A reviewis given by the registered user based on their personal experience with the course and any advice they may have. This will be an optional input. 

3.17 Search: Searchallows the user to search for a course using the course code or course name.

3.18 Course Filter: A filter system that allows the user to filter the courses. Some possible filters include course level, the minimum number of ratings, the term offered, and prerequisites.

3.19 Desirability Filters: A filter system that allows the user to filter the courses based on the desired level of enjoyability, ease, and usefulness. For example, if students would like to view the easiest courses offered at WLU, they can set the birdness filter to high.

4.2 Internal Structure

In future phases, the team structure will differ to provide further organization and structure.

Documentation

    1. Authors: authors are in charge of writing the rough draft of their respective document and must ensure that their work aligns with client expectations. They will finish their work early to ensure that the SQA team has enough time to make adjustments. 
    2. Management: management is in charge of creating meetings, ensuring that each team is on schedule to meet deadlines, and facilitating coordination between the different teams. Additionally, each team in the implementation phase will have a manager that is to ensure coordination between the front end, back end, and SQA team. They will also set up team meetings, assign tasks, and ensure deadlines are being met. 
    3. Software Quality Assurance (SQA): ensures high-quality work that aligns with the client’s expectations. They must also ensure that there is consistency throughout the document. They can create test cases and enter adjustment suggestions into the SQA revision history for team members to check and update accordingly.
    4. Version Control: version control ensures that the documentation is properly organized and converted to HTML.

Implementation

This subclause of the SPMP shall describe the internal structure of the project organization to include the interfaces among the units of the software development team. In addition, the organizational interfaces between the project and organizational entities that provide supporting processes, such as configuration management, quality assurance, and verification and validation, shall be specified in this subclause. Graphical devices such as organizational charts or diagrams should be used to depict the lines of authority, responsibility, and communication within the project.

5.2 Schedule Allocation

This section will specify the procedures necessary to ensure the timely progression of the project. The team as a whole will meet each week at a mutually agreed-upon time. All communication is to take place on Discord where each team has its own channel. In each meeting, each person will provide an update on the work they have done and discuss any problems they have faced. In each phase, there will be 3 members assigned to the management role. These managers will equally divide the work required for the phase and lead their sub-team in completing their section of the phase. Managers will schedule meetings with their teams, ensure tasks are appropriately delegated and completed on time. They will also communicate regularly with other managers to ensure consistency across the 3 sub-teams and update the team as a whole regularly on any problems encountered. 

5.3 Monitoring Metrics

Each member will enter in the amount of time spent working on each phase in the shared Google Sheets along with a short description of their contributions. There will also be an SQA review of each phase. Each member on the SQA team will report problems/recommendations in the shared Google Sheets along with a priority level. The members who worked on the deliverable will refer back to the SQA and make the appropriate adjustments. There will be a shared repository via Github.

6. Techical Process Plans

6.1 Process Model

To ensure the successful completion of the project, the tasks have been divided per the strengths of various team members. Project progress is categorized into two main areas: Documentation and Implementation. The documentation portion aims to put in writing the facets of the project and provides more information and guidance on implementation. 

The Implementation portion will focus on creating a proof of concept of the ideas outlined in the documentation. 

Documentation

  • Authors 
    • Responsible for the initial write-ups of documentation and process definition.
  • SQA
    • Assess the work done by the authors to ensure that the quality standards have been met, spell check, and ensure that the documentation is ready for submission. 
  • Version Control 
    • Deploys the final HTML version of the documentation and ensures that it is legible 
  • Revision Team 
    • Responsible for making updates to documentation that has already been finalized. 
  • Team Leads 
    • Act in a management capacity to ensure that deadlines for project phases and submissions are met. Additionally, they update the deadlines and ensure participation from all team members.

Implementation

As mentioned above the division of tasks among members is based on their skillsets. 

  • Full Stack 
    • Members work on both frontend and backend components and will be relied on to make sure that both components integrate well and function properly. 
  • Frontend 
    • Works on the User Interface/User Design and ensures they meet the specifications stated in the requirements. 
  • Backend 
    • Builds the back-end infrastructure of the application and ensures that it meets the requirements stated in the specification document.
  • SQA
    • Responsible for testing the application on top of what tests were already performed by the individual teams. 
  • Team Leads 
    • Ensure that the team meets the deadlines and all team members contribute towards the completion of the project.

6.2 Methods, Tools & Techniques

This outlines the agreed-upon methods and techniques that are to be followed till the completion of the process. 

  • Communication 
    • The team will use discord to communicate and ensure that we stay on track. For the various teams such as frontend, backend, and full-stack specific channels have been created and each will have the autonomy to set up their own meetings.
  • Tools 
    • IDE
      • The teams will be using VS Code as their editor and can use any external plugins that will allow them to code optimally. 
      • Documentation in the code will be done using hashtags and more detailed ones will be done using multiline strings. 
      • All functions/methods in the backend will be written in multiline strings and will contain the aim of the function, the arguments, and the expected returns.
    • Git
      • Each time will have a specific branch to deploy their code
      • The commits made will need to meet the standards defined by the teams
  • Database 
    • To help with designing the database schema external tools such as dbdiagram.io will be utilized. 
    • The database used will be MYSQL and will be deployed to a cloud provider such as Google Cloud/AWS/Azure. 
    • For database management such as managing external access to the database and querying the database, the backend team will use MYSQL workbench.

7. Supporting Process Plans

7.1 Verification Plans

The verification and validation process focuses on ensuring that the standards set are consistently observed. This verification process is what the SQA process is intended to achieve. 

Verification will involve ensuring:

  • Accuracy 
    • This process ensures that the final product at each phase of the project lines up with what was detailed in the requirements documentation of the project.
  • Up-to-date Documentation
    • Consistent update to the documentation to include new features, remove features that cannot be delivered or provide more comprehensive features regarding certain features. The details in the documentation will be used to assess the success of the final product hence it is essential to have it be as accurate as possible.
  • Functionality 
    • This portion involves ensuring that the code meets the functional requirements that were drafted. For this to be achieved the frontend and backend teams will have to perform functionality testing and will need to test their code to ensure that no syntax or logic errors are embedded.
    • Additionally, team members that did not take part in writing the code will also test it thus completing the code quality assurance cycle.

7.2 Problem Resolution

As noted above the management team will be responsible for ensuring that the deadlines are met. In case any issues arise below are the potential avenues to solve the problems:

  • Re-allocating tasks to ensure that they are completed on time. 
  • Allowing more time for individuals to complete the assigned parts of the project/assignments. 
  • Following up with team members on the progress of the tasks that they need to do. 
  • Scheduling meetings whether general or between the specific teams to discuss progress or changes to specification and documentation.

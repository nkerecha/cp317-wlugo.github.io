    1. Introduction

    In this section, an overview of the WLU Go Software Design Document will be given.

    1.1 Purpose

    The Software Design Document provides a general description of the WLU Go software architecture as well as major policy decisions. The document will describe how the requirements in the SRS will be achieved by providing detail on the required system sources, data, and functional structure to be implemented.

    1.2 Audience

    The intended audience of this document is developers tasked with the creating, testing, and maintaining the WLU Go web application.

    1.3 Scope

    The scope of the project is a fully functional web-based application that allows users access to student-given reviews on every course offered at Wilfrid Laurier University.

    1.4 Definitions

    1.4.1 WLU: Wilfrid Laurier University. The educational institution whose courses will be evaluated by the users.

    1.4.2 API: Application Program Interface. External software packages are the intermediary between applications used in building WLU Go.

    1.4.3 GUI: Graphical User Interface. This is the interface with which the user interacts. It consists of traditional navigation buttons, rating bars, and relevant information.

    1.4.4 SRS: Software Requirements Specifications. This is the document that has all requirements and standards contained.

    1.4.5 DDMS: Database Management System. This is the software that is used to store and retrieve users’ data while considering appropriate security measures.

    1.4.6 FMS: File Management System. This is the process of administering a system that correctly handles all digital data/information/files.

    1.4.7 HTTPS: Hypertext Transfer Protocol Secure: Hypertext Transfer Protocol Secure is an extension of the Hypertext Transfer Protocol. It is used for secure communication over a computer network and is widely used on the Internet.

    1.4.8 Birdness: The level of course difficulty experienced by the student reviewer on a scale of 1-5 with 1 being very difficult and 5 being very easy.

    1.4.9 Usefulness: The level of course usefulness and applicability experienced by the student reviewer in their field on a scale of 1-5 with 1 being very useless and 5 being very useful.

    1.4.10 Enjoyability: The level of course enjoyability experienced by the students of 1-5 with 1 being very unenjoyable and 5 being very enjoyable.

    1.4.11 Grade Received: The grade that the user received after completing the course. This will be an optional input for all users. 

    1.4.12 Professor: The full name of the professor who taught the course. This will be an optional input for all users. 

    1.4.13 Course Name/Course Code: The course code or the name of the course.

    1.4.14 Course Description: The description of the course that Wilfrid Laurier University provides.

    1.4.15 Guest User: A guest user is someone who uses the application to view reviews. 

    1.4.16 Review: A review is given by the registered user based on their personal experience with the course and any advice they may have. This will be an optional input. 

    1.4.17 Search: Search allows the user to search for a course using the course code or course name.

    1.4.18 Course Filter: A filter system that allows the user to filter the courses. Some possible filters include course level, the minimum number of ratings, the term offered, and prerequisites.

    1.4.19 Desirability Filters: A filter system that allows the user to filter the courses based on the desired level of enjoyability, ease, and usefulness. For example, if students would like to view the easiest courses offered at WLU, they can set the birdness filter to high.

    2. System Overview

    WLU Go is a web application that provides students a digital space to view and give feedback on all courses offered at Wilfrid Laurier University.

    2.1 Description

    At the top level, WLU Go is a web application that is split into three key main sections. There will be the key system functionalities, key system components and the key UI components.

    1. Key System Functionalities
      1. Searching for courses.
      2. Posting reviews of courses.
      3. Viewing courses and their reviews.
    2. Key System Components
    1. Front-end.
    2. Back-end.
    1. Key Front-end Pages (UI), their Components and their functions
    1. Home page: 
      1. Header: containing logo and navigation buttons include signup and login buttons.
      2. Search bar: to search through a dynamic dropdown course list, each course row including course code and title. Query is either course code or title.
      3. Footer: containing About page, Privacy page, Instagram page, and Email redirect.
    2. Course page: 
    1. Header: containing logo, search bar specified below, logo, redirecting to home page, and user profile badge specified below. 
    2. Search bar: to search through a dynamic dropdown course list, each course row including course code and title. Query is either course code or title.
    3. Course banner: containing metrics (birdness, usefulness, enjoyability, score), description, code and other potential relevant information.
    4. User profile badge: user logo and a dropdown to sign out.
    5. List of reviews: 
      1. Review cards containing the review, all metrics (birdness, usefulness, enjoyability), the date posted, and optional information such as the grade the reviewer received in the course.
    6. Footer: containing About page, Privacy page, Instagram page, and Email redirect.           
    1. Login page:
      1. Header: containing the logo.
      2. Login form including email, password, forgot password and login button.
      3. Footer containing About page, Privacy page, Instagram page, and Email redirect.
    2. Signup page:
      1. Header: containing the logo.
      2. Sign up form including first and last name, email, password, confirm password, program, graduation year and sign up button.
      3. Footer containing About page, Privacy page, Instagram page, and Email redirect.

    2.2 Key System Components

    The two major components of the WLU Go web application will be the front-end and back-end.

    2.2.1 Front-end Design

    The front-end of the web application will have these key functions and use these languages/technologies:

    • Populate the UI/UX with data that is requested from the back-end.
    • Assemble the course pages with the appropriate UI subcomponents (course title, course rating circle, course metrics bar, etc.).
    • Request the data from the endpoints of the program to populate the subcomponents with their respective data.
    • The front-end will be done using  HTML, CSS and React JavaScript.

    2.2.1 Back-end Design

    The back-end of the web application will have these key functions and use these languages/technologies:

    • Fetch all of the data/input that is requested from the front-end.
    • Store all of the data that is entered from the user into the database.
    • The back-end will be done using Python, Node JS and MySQL for the database.
    • The web and API hosting will be done AWS (Amazon Web Services).

    2.2.3 Communications Mechanisms

    The front-end and back-end components will communicate via RESTful API calls written in JavaScript.

    3. Design Considerations

    3.1 Assumptions and Dependencies

    • Able to be used on all modern operating systems and web browsers.
    • The users have a stable internet connection while on the site.

    3.2 End-user Characteristics

    • Users are able to search for courses, view the description, metrics and reviews posted, as well as write and post reviews about a course. 
    • Moderators are given increased privileges such as removing certain posts reported by other users, banning users.
    • Administrators are given the most privileges: all moderator privileges and the ability to assign and strip moderator privileges from users.

    3.3 General Constraints and Limitations

     

    • All users must have a valid email and a secure password for security authorization.
    • All screens must be above 320px in order to meet usability requirements.

    3.4 Goals and Guidelines

    • Functionality shall be prioritized over design until functionality outlined is achieved.
    • UI must be modern and minimal. 
      • Follows a precise colour scheme (see 9.1.1).
      • Functional and accessible, easy to operate (see 9.2).
      • Interface optimised for accessibility, readability and functionality. (see 9.1.2).
      • Users will be able to easily understand the purpose of the website.
    • All code must be readable, functional, minimize repetition, and include necessary comments.
    • Any contribution made should be documented in the tracking sheet.
    • The security of user confidential data shall never be compromised, adhering to the privacy policy that will be presented on the website.

    3.5 Developement Methods

    • Initial design was designed by the design sub-team using Figma.
    • Analysis and certain aspects of the project were revised many times in order to achieve the most efficient iteration and receive the best possible outcome.
    • Split the team into three sub teams being full-stack, front-end, and back end teams to ensure everyone is aware of their tasks.
    • Agile methodology will be used in Scrums and managed using GitLab free trial.

    4. Design Decisions, Policies, and Tactics

    4.1 Deployment

    The WLUGo web application will be constructed using REACT.JS an open source, front end Javascript library that is used for building User Interfaces and web applications. It will be coded in Javascript, HTML, CSS for the front end, alongside Python and Node.JS for the backend. Due to the large databases needed to manage/store all the course data, MySQL will be used as the database.

    4.2 Data Storage

    For the Data Storage the team will be using MySQL on a server hosted and managed by AWS (Amazon Web Services).

    4.3 Coding Guidelines and Conventions

    • The front end team will have a front end team primarily using Javascript on REACT alongside HTML and CSS.
    • The back end team will write and test the code in Node.JS and python along with creating and managing the database in MySQL.
    • The full stack team will be in charge of fully incorporating the back end Node.JS and python code alongside the front end design with Javascript and making sure everything is working on both ends.

    4.4 Ensuring Requirements Traceability

    Requirement traceability will be ensured using a traceability matrix.

    4.5 Software Testing

    Software will be regularly tested by the WLU Go team after every update alongside a weekly full thorough run through to ensure everything is working and up to date. 

    During the final stages of implementation extensive debugging and software testing will take place to ensure WLU Go is ready to go for launch. Github will be used for version control and whitebox testing will be handled by the team.

    4.6 Software Maintenance

    The software will be maintained by the WLU Go team until August 31st 2021.

    5. System Architecture

    5.1 System Architecture Description

    Home is the central interface where the user can access the rest of the pages available. It also acts as the search where users can access the specific courses on the course page. Sign in and user profile can be accessed through the course interface, but remain independent of each other.

    5.2 System Architecture Diagram


    6. Architectural and Component-level Design

    The WLU Go web application will have 5 key classes, each with unique functions. There will be a User, Review, Course, Courses and Rating class. Each class will also have its own attributes, types, constructors, getters and setters.

    6.1. Class Diagram

    Class Diagram Clarifications

    • The Review classes’ body attribute is the optional text a user can enter in a review. If no body is entered, a review post won’t be created, rather a rating will be left. 
    • The User class has a method called deleteUser that takes an User object as a parameter. A user can only delete their own account, but admins and moderators can delete other users accounts. 
    • The User class attribute hasAccess is false if a user has been banned and true if not. A user is banned if a moderator calls the banUser(user : User) method passing the banned User object in as a parameter.
    • The User class contains attributes that determine whether a user is an admin, a mod or a regular user. A regular user will hold false values for the isMod and isAdmin attributes. The methods under <<assert this.isMod>> may only be accessed if the user is a moderator and the methods under <<assert this.isAdmin>> may only be accessed if the user is an admin. Mods can ban/delete regular users and delete reviews. Admins have all moderator permissions and can also give and take admin and moderator permissions to and from users.  An admin can call the setIsAdmin( isAdmin : Boolean, user : User ) method, which will take the User object and call setAdminByAdmin( isAdmin : Boolean, admin : User), where admin is the admin object to prove that this operation is being done by an admin. The same logic applies to the setIsMod( isMod : Boolean, user : User ) function.

    7. Data Design

    7.1 Data Structures

    This web application will use Global Data structures to retrieve all information globally.

    7.1.1 Global Data Structure

    In order to access information in the many components of WLU Go, a global database will be created. Additionally, each table in the database has its own class to represent the data globally.

    7.1.2 Database Description

    Upon sign up, general users will be given a unique user ID that will be used to link to the users’ details and each course review they provide. The courses table stores information about each course offered at WLU and has a unique key that ties a user’s review to the corresponding course code.

    7.2 Database Diagram

    7.3 Database Tables

    7.3.1 RATING Table

    RATING

    Contains properties with specified rating metrics. 

    • rating_id, int - The unique ID assigned to a review object 
    • birdness, int (1 - 5) - The easiness rating of a course from 1 being very hard to 5 being very easy.  
    • usefulness, int (1 - 5) - The usefulness rating of a course from 1 being utterly useless to 5 being very difficult.
    • enjoyability, int (1 - 5) - The enjoyability rating of a course from 1 being extremely unenjoyable to 5 being very enjoyable. 

    Relationships

    The rating_id is the primary key, it has a one-to-one relationship with the COURSE and REVIEW tables.

    7.3.2 USER Table

     

    USER

    Contains records of users that used to sign up as well as more user details.  

    • user_id, int - Unique id assigned to the user when they signed up.
    • email, varchar (320) - The email provided by the user during signup.
    • password, varchar (255) - The password provided by the user during signup.
    • first_name, varchar (255) - The first name provided by the user during sign up.
    • last_name, varchar (255)  - The last name provided by the user during sign up.
    • program, varchar (255) -The program that the student is enrolled in. 
    • graduating_year, int - The users expected graduation year. 
    • has_access, boolean - The user’s access to the web application, if false, user is banned. 
    • is_mod, boolean - The user’s moderator privileges, if true, the user is a moderator. 
    • is_admin, boolean - The user’s administrator privileges, if true, the user is an admin.. 

    Relationships

    The user_id is the primary key in this table and has a one-to-many relationship with the REVIEW table.

    7.3.3 COURSE Table

    COURSE

    contains a record for every course at WLU

    • course_id, int - The unique ID associated with a course.
    • code, varchar (255) - The unique course code (e.g. CP104) associated with the course.
    • name, string - This is the name associated with a course code (e.g. Software Engineering). 
    • description, string - Details that explain what a specific course at WLU explains.
    • department, string - Department that a particular course is under.
    • rating_id, int - The rating_id of the rating object holding the average birdness, usefulness, and enjoyability of all review ratings.
    • score, int (0-100) - The aggregate score calculated by the average of rating metrics specified in the rating table.
    • median_grade, int (0-12) - The median grade from all reviews.

    Relationships
    • The course_id is the primary key of the COURSE table and has a one-to-many relationship with the REVIEW table and a one-to-many relationship with the PROFESSOR_COURSE table.
    • The rating_id has a one-to-one relationship with the RATING table.

    7.3.4 REVIEW Table

    REVIEW

     Contains a record for every review that is made

    • review_id*, int - A unique ID that is attached to every review that is made.
    • user_id, int - The user_id that is attached to a logged in user that left a particular review.
    • course_id, int - The course id that a particular review is posted on.
    • body, text - The text that a user leaves on a review.
    • rating_id, int - The rating_id assigned to a review object.
    • grade, int - An integer value ranging from (1 - 12) which represents the grade the user received for the course.
    • professor_id, int - The professor_id associated with the professor that taught the course at the time the student attended.  
    • term_taken, varchar (255) - The semester and year a course was taken.
    • date_time_created, datetime- The date and time that the review was made by the user.

    Relationships
    • The review_id is the primary key of the REVIEW table.
    • The user_id is the foreign key that has a many-to-one relationship to the USER table.
    • The course_id is the foreign key that has a many-to-one relationship to the COURSE table.
    • The rating_id is the foreign key that has a one-to-one relationship with the RATING table.
    • The professor_id is the foreign key that has a one-to-one relationship with the PROFESSOR table.

     

    7.3.5 PROFESSOR Table

    PROFESSOR

    Contains a record for every professor. 

    • professor_id, int - The unique id that identifies a professor. 
    • name, varchar (255) - The full name of the professor.  

    Relationships

    The professor_id is the primary key with a one-to-many relationship with the PROFESSOR_COURSE table.

    7.3.6 PROFESSOR_COURSE Table

    PROFESSOR_COURSE

    Seperate table required since a course can have many professors and a professor can teach many courses. 

    • professor_id, int - The foreign id that identifies a professor.  
    • course_id, int - The foreign id that identifies a course.  

    Relationships
    • The professor_id has a many-to-one relationship with the PROFESSOR table.
    • The course_id has a many-to-one relationship with the COURSE table.

    8.1.2 Font Guidelines

    The main font-family that will be used is Montserrat. Bold and extra-bold will be used for headings and subheadings and regular will be used for paragraph text. The size of the headings should be larger than subheadings which should be larger than paragraph text. These font sizes will be responsive to allow readability on web and mobile browsers.

    8.2 User Interface Views

    The homepage will consist of a search bar and a short description/describing WLU Go and what it has to offer. One can interact with the search bar on the home page by typing in the code or course name they are interested in. Once they type in the specific WLU course code or name, a drop down list of courses will be displayed with clickable courses that will navigate them to the respective course page. This is shown visually in the sample GUI of the home page (found in section 9.2.1). Each WLU course will have its own unique course page. This page will display a variety of different quantitative and qualitative data in an understandable and simple manner. From the course page, a user will be able to view the birdness, usefulness, enjoyability, overall score and average grade of a course presented as an aggregate of all the course reviews provided by other users of the site. In addition, users will be able to see comments left behind of other students going into detail about their experiences with the respective courses. This is shown visually in the sample GUI of the course page (found in section 9.2.2).
